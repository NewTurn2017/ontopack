명령 치트시트 문서 지도 다음 우선순위 README ↗
local-first · portable · agent-ready

로컬 지식을 이동 가능한 팩으로 만든다.

OntoPack은 Markdown, assets, 검색 인덱스, citation-ready context, 미디어 enrichment를 한 폴더에 묶는 Rust 기반 OSS 지식 팩 엔진입니다. 이 페이지는 README보다 더 빠르게 전체 구조를 이해하도록 만든 wiki형 안내서입니다.

Plain filesnotes/*.md + assets/*가 원본
SQLite cache검색 DB는 재생성 가능한 파생물
MCP-readyAI 에이전트에 citation context 제공
CI provenmacOS · Linux · Windows smoke

왜 필요한가

노트 앱, 벡터 DB, 미디어 폴더, 에이전트 메모리가 따로 움직이면 지식은 금방 깨집니다. OntoPack은 사람이 읽는 파일을 원본으로 두고, 검색/뷰어/MCP/번들은 모두 그 원본에서 다시 만들 수 있게 설계합니다.

학습/강의

자료, 이미지, 영상, 소스 카드를 하나의 강의 팩으로 관리합니다.

에이전트 컨텍스트

LLM에게 답을 맡기기 전에 citation-ready context block을 안정적으로 넘깁니다.

운영 가능한 OSS

Windows smoke, release workflow, test evidence로 “내 컴퓨터에서만 됨”을 줄입니다.

핵심 모델

OntoPack 팩은 특별한 서버가 아니라 디렉터리입니다. 원본과 파생물을 분리해서, 사람이 편집해도 다시 빌드할 수 있습니다.

1. Capture_inbox/에 노트/미디어 투입
2. Processnotes/, assets/로 정리
3. Build.pack/index.db 재생성
4. Use검색, 뷰어, MCP, AI worker
5. Moveexport/bundle/import로 이동
my-pack/
├─ pack.toml              # 팩 설정
├─ notes/*.md             # 사람이 읽고 수정하는 지식 원본
├─ assets/*               # 이미지/영상/오디오/PDF 등 원본 파일
├─ _inbox/*               # 임시 수집함
└─ .pack/index.db         # 재생성 가능한 검색 캐시

5분 체험

설치 후 아래 명령만으로 “노트 생성 → 처리 → 검색 → 로컬 wiki 열기 → 이동 가능한 번들”까지 확인합니다.

pack init ~/ontopack-demo
cd ~/ontopack-demo

cat > _inbox/hook.md <<'NOTE'
---
type: prompt
title: 썸네일 훅
tags: [youtube, hook]
created: 2026-05-24
---
클릭을 부르는 훅 카피와 강의 오프닝 구조.
NOTE

pack process
pack build --incremental
pack search "훅" --mode keyword
pack open
pack bundle ../ontopack-demo-bundle --archive ../ontopack-demo.tar.gz
성공 기준: 검색 결과에 [keyword], note_id, chunk_id가 보이고, pack open에서 로컬 뷰어가 열리면 기본 경로가 살아 있습니다.

설치

macOS / Linux source build

git clone https://github.com/NewTurn2017/ontopack.git
cd ontopack
cargo build --release -p pack-cli
./target/release/pack --help
scripts/install.sh --prefix "$HOME/.local" --completion-shell zsh
pack --version
pack setup --shell zsh
pack doctor

전체 workspace 빌드

cargo build --release
./target/release/pack-mcp --help
cargo test --workspace --locked

공개 tagged binary release는 아직 첫 태그를 자르기 전입니다. release workflow는 준비되어 있으므로 태그 발행 후 macOS/Linux/Windows archive와 checksum이 생성되는지 확인하면 됩니다.

Windows 경로

OntoPack은 pack-relative path를 forward slash로 유지하고, Rust path join으로 OS별 파일 시스템에 맞춥니다. Windows CI는 기본 CLI test와 smoke를 실행합니다.

git clone https://github.com/NewTurn2017/ontopack.git
cd ontopack
cargo build --release -p pack-cli
.\target\release\pack.exe --help
powershell -ExecutionPolicy Bypass -File .\scripts\windows-smoke.ps1 -PackBin .\target\release\pack.exe
Provider worker 팁: Windows에서 Python .py를 직접 실행할 수 없으면 --provider-command py --provider-arg scripts\providers\auto_media_worker.py처럼 Python launcher를 명시합니다.

릴리스/CI

현재 운영 기준선은 .github/workflows/ci.yml과 .github/workflows/release.yml입니다. CI는 formatting, tests, clippy, release binary build, Unix MVP smoke, Windows portability smoke를 실행합니다.

# 로컬에서 push 전 최소 검증
cargo fmt --check
cargo test -q
cargo clippy --all-targets -- -D warnings
scripts/mvp-smoke.sh
scripts/real-test.sh

검증 루틴

기능을 추가할 때는 “명령이 실행된다”보다 “사용자가 신뢰할 수 있는 증거가 남는다”를 기준으로 봅니다.

  1. 작은 변경: 관련 단위 테스트 + cargo fmt --check.
  2. CLI 동작 변경: cargo test -q + cargo clippy --all-targets -- -D warnings.
  3. 운영/릴리스 변경: smoke script, workflow YAML parse, GitHub Actions run URL까지 기록.
  4. 문서 변경: 링크, 코드 블록, 실제 명령 이름이 현재 CLI와 맞는지 확인.
  5. 설치/초기 설정: pack --version, pack setup, pack doctor를 순서대로 실행해 사용자가 다음 행동을 알 수 있는지 확인.

캡처와 처리

pack add는 바로 추가하고, pack process는 _inbox/를 한 번에 정리합니다. Markdown/text는 notes/, 바이너리 파일은 assets/와 sidecar note로 들어갑니다.

pack add ~/Downloads/source.md --type note
pack add ~/Downloads/board.png --type image
pack process
pack status --json
pack list --pending-enrichment

검색과 인덱스

기본 검색은 SQLite FTS5 keyword search입니다. 인덱스는 notes/와 enrichment block에서 재생성할 수 있으므로 안전하게 삭제/재빌드할 수 있습니다.

pack build --incremental
pack search "ontology" --mode keyword -k 5
pack duplicates
pack gaps
pack topics
pack recommend graph-primer

semantic/vector/hybrid는 real-embed feature로 빌드한 뒤 pack embed를 먼저 실행해야 합니다.

로컬 뷰어

pack open은 별도 프론트엔드 빌드 없이 binary가 localhost wiki/gallery/search UI를 서빙합니다. 원격/자동화 환경에서는 URL만 출력할 수 있습니다.

pack open
pack open --port 0 --no-browser --print-url
pack serve --host 127.0.0.1 --port 0
좋은 OSS 데모: README만 보여주기보다 샘플 팩을 열어 검색, 갤러리, 그래프, source card를 한 화면에서 보여주면 이해 속도가 빨라집니다.

번들/복원

OntoPack의 운영 강점은 “다른 머신으로 옮겨도 다시 검색 가능한 팩”입니다. directory bundle과 tar.gz archive를 모두 지원합니다.

pack export --format jsonl --output context.jsonl
pack bundle ../demo-bundle --archive ../demo.tar.gz

pack init ../restored-pack
cd ../restored-pack
pack import ../demo.tar.gz
pack build --no-embed
pack search "ontology"

MCP 서버

pack-mcp는 OntoPack을 Claude/Codex 같은 에이전트가 호출할 수 있는 stdio MCP 서버로 노출합니다. 중요한 점은 서버가 답변을 생성하지 않고 citation-ready context blocks를 반환한다는 것입니다.

cargo build --release -p pack-mcp
pack-mcp --pack-root /path/to/my-pack
도구용도
search, ask검색 결과와 citation context block 반환
related, timeline관계 탐색과 created frontmatter 기반 시간순 탐색
media/list_pendingAI caption/OCR/transcript가 필요한 asset sidecar 탐색
media/write_enrichment사람이 쓴 본문을 보존하며 managed enrichment block 기록

Provider workers

Provider worker는 stdin으로 payload JSON을 받고 stdout으로 enrichment patch JSON을 쓰는 실행 파일입니다. 저장 형식은 OntoPack이 지키고, 모델/도구 선택은 worker가 담당합니다.

# deterministic fixture worker
pack enrich-pending --provider-command scripts/providers/fixture_media_worker.py --limit 10

# Windows Python launcher shape
pack enrich-pending --provider-command py --provider-arg scripts\providers\fixture_media_worker.py --limit 10

# recommended router
ONTOPACK_PROVIDER_MODE=local pack enrich-pending --provider-command scripts/providers/auto_media_worker.py --limit 1

지원 필드: caption, tags, ocr, transcript, summary, keyframes, provider, model.

real-embed

기본 설치는 offline keyword-first입니다. semantic vector/hybrid search가 필요할 때만 feature를 켭니다.

cargo build --release -p pack-cli --features real-embed
./target/release/pack embed
./target/release/pack search "ontology" --mode hybrid
운영 메모: 첫 real-embed 실행은 FastEmbed/Hugging Face 경로로 모델을 다운로드/캐시할 수 있습니다. CI 기본 경로와 분리해두는 편이 안전합니다.

보이는 온톨로지 팩

OSS 설명은 실제 데이터가 보이면 훨씬 빨라집니다. OntoPack은 공개 웹 API 메타데이터로 prompt/image/video 각 100개를 가진 demo pack을 만들 수 있습니다.

cargo build -p pack-cli
python3 scripts/visible-ontology-pack.py \
  --limit-each 100 \
  --no-download-assets \
  --output /tmp/ontopack-visible-live \
  --pack-bin "$PWD/target/debug/pack" \
  --build
cd /tmp/ontopack-visible-live
/path/to/ontopack/target/debug/pack open --no-browser --print-url

다음 우선순위

  1. 첫 public release tag: release workflow가 산출하는 archive/checksum을 실제로 내려받아 smoke합니다.
  2. 문서 사이트 분화: 이 wiki형 landing을 기준으로 설치, 튜토리얼, MCP, provider, 운영 runbook을 독립 HTML 페이지로 나눕니다.
  3. 샘플 팩 배포: fixture 기반 demo pack을 release artifact나 docs asset으로 제공해 “보이는 이해”를 강화합니다.
  4. 서명/배포 안정화: macOS signing/notarization, Windows signing은 아직 남은 운영 과제입니다.
